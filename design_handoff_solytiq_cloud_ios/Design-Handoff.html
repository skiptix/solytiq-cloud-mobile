Solytiq Cloud — iOS

This is the complete design + build reference for the Solytiq Cloud iOS UI kit — a mobile-native rendition of the "Luminous List" design language, built as a self-contained React prototype at ui_kits/solytiq-cloud-ios/. Use this doc to hand the kit to designers (visual language, spacing, states) and developers (component API, file structure, tokens) without needing to reverse-engineer the source.

Design language: Luminous List Platform: iOS / mobile web Status: Prototype — ready for native handoff

File Map & Setup

The iOS kit is a standalone folder, independent from the desktop/web kit (ui_kits/solytiq-cloud/). Every file below lives at ui_kits/solytiq-cloud-ios/.

index.html
Document shell — CSS custom properties, keyframes, script load order, seed data, root <App/>, Tweaks wiring.
components.jsx
Core primitives: SFSymbol, PhoneFrame, NavHeader, TaskRow, QuickAdd, StatCard, Card, SectionHeader, EmptyRow, Badge, FileBadge, StorageBar, FloatingTabBar, AddFloatBtn, ProfileBtn, SideRail.
sheets.jsx
Bottom-sheet modals: task detail/edit, add-choice, add folder/list, lists drawer, settings, AI assistant, trash, file preview, filters.
screens.jsx
Full-page screens: Welcome, Login, Dashboard, List, Scheduled, Files, Folder dashboard, Lists index, plus the AppShell router.
features.jsx
Extended feature set: Timelines (list + detail), workspace switcher/wizard, item-settings, 2FA setup sheet.
calendar.jsx
Unified calendar screen (tasks + meetings + milestones) and its add/filter sheets.
ios-frame.jsx
A second, reusable "iOS 26 Liquid Glass" device-frame kit (status bar, glass nav pills, grouped lists, keyboard) for static mockups outside the live prototype.
tweaks-panel.jsx
Generic Tweaks-panel host protocol — palette / shape / density controls.

Running it: open index.html directly — no build step. React 18 + Babel Standalone are loaded from unpkg with SRI hashes; every other file is a plain <script type="text/babel"> include, executed top-to-bottom in the order listed above.

Frontend Architecture

No bundler, no JSX build step — this is an in-browser-transpiled prototype, intended to be the interaction/visual spec that a native iOS (SwiftUI) or React Native build should match pixel-for-pixel, not shipped as-is to production.

Global namespace pattern

Every file assigns its exports onto window instead of using ES modules, so plain <script> tags can share components without a bundler:

Object.assign(window, {
  SFSymbol, PhoneFrame, NavHeader, TaskRow, /* … */
});

State lives in one root App component in index.html (seed data for tasks / lists / folders / files / meetings / workspaces / timelines), passed down through a single AppCtx React context, read anywhere via window.useApp(). Screens/sheets read and mutate this context directly — there is no separate store library.

Screen & sheet routing

AppShell (bottom of screens.jsx) is the router: app.screen selects the current full-screen view from a lookup map, app.modal selects the active bottom sheet from a second lookup map. Setting either is as simple as app.setScreen('list') or app.setModal('edit-task') — both exposed on the shared context.

Device Frame

Two independent device-frame implementations exist — pick the right one for the job.

9:41

PhoneFrame (components.jsx)

  • 393 × 852px outer, 9px bezel, 56px corner radius — modeled on iPhone 15/16.
  • Custom-drawn status bar (time + signal/wifi/battery SVGs), 126×36 dynamic-island pill, 134×5 home indicator.
  • Used to wrap the live <AppShell/> for the interactive prototype — this is what you see in index.html.
  • Background reads --sc-page so Tweaks re-color it live.

IOSDevice (ios-frame.jsx)

  • 402 × 874px default, 48px radius — a fuller "iOS 26 Liquid Glass" kit: real blur/glass nav pills, grouped inset lists (IOSList/IOSListRow, 26px radius, 52px rows), and a full QWERTY IOSKeyboard.
  • Standalone / dependency-free — reach for this when mocking a single static iOS screen outside the running prototype (e.g. App Store screenshots, spec panels).
  • Supports a dark prop for light/dark chrome variants.

Navigation Chrome

Chrome pieceBehavior

Screen transitions are directional slides driven by a rank order (SCREEN_RANK in AppShell) — moving to a "later" screen slides in from the right, moving "back" slides in from the left, matching iOS navigation-stack conventions even though this is a flat tab structure under the hood.

Dev Handoff Notes

For native (SwiftUI / RN) engineers

Treat this prototype as the interaction and pixel spec, not shippable code:

  • All spacing/radius/color values above are exact — port them as design tokens in your native token layer rather than eyeballing from screenshots.
  • Spring curves used throughout are cubic-bezier(0.34, 1.56, 0.64, 1) (overshoot spring) and cubic-bezier(0.32, 0.72, 0, 1) (screen slide) — map to UISpring/SwiftUI .spring() equivalents.
  • The large-title collapse behavior in NavHeader matches UINavigationBar's native large-title-to-inline transition — implement with the platform-native component where possible rather than recreating scroll-driven opacity by hand.
  • The bottom sheet system (sheetIn, 26px top radius, drag handle, backdrop blur) maps directly to UISheetPresentationController / SwiftUI .sheet with a custom detent.

For designers extending the kit

  • New screens should register in SCREENS and SCREEN_RANK in AppShell (screens.jsx) to get transitions and chrome visibility for free.
  • New sheets register in the SHEETS map in the same file — no extra plumbing needed, the sheet chrome (backdrop, handle, radius) is automatic.
  • Reuse Card / SectionHeader / TaskRow / EmptyRow before building new list patterns — nearly every screen composes from these four.

Scope note: this is a front-end-only prototype with in-memory seed data (see SEED_* constants in index.html) and localStorage for profile persistence only. No network layer exists in this kit — API wiring follows the contract already defined for the web app in the top-level README.md.
